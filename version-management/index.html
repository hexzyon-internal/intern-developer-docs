VERSION File Management

Complete guide to semantic versioning, VERSION file workflows, and Docker image tagging for our CI/CD pipeline.

8 min read Updated Dec 14, 2025 Pasindu Jayasinghe

VERSION File Format

The VERSION file should contain only the semantic version number in a clean, standardized format.

1.0.0

Formatting Rules

  • No 'v' prefix in the file
  • No extra whitespace or line breaks
  • Follow semantic versioning: MAJOR.MINOR.PATCH
  • Single line only

Version Bump Decision Tree

Did you make breaking changes?
├─ YES → Bump MAJOR (1.0.0 → 2.0.0)
└─ NO
   ├─ Did you add new features?
   │  └─ YES → Bump MINOR (1.0.0 → 1.1.0)
   └─ NO
      └─ Did you fix bugs?
         └─ YES → Bump PATCH (1.0.0 → 1.0.1)

Common Version Update Scenarios

Scenario 1: Bug Fix

Before: 1.2.3

After: 1.2.4

echo "1.2.4" > VERSION
git add VERSION
git commit -m "fix: resolve authentication issue"
git push origin sync

Result: Image tagged as ghcr.io/username/project:v1.2.4

Scenario 2: New Feature

Before: 1.2.4

After: 1.3.0

echo "1.3.0" > VERSION
git add VERSION
git commit -m "feat: add user dashboard"
git push origin sync

Result: Image tagged as ghcr.io/username/project:v1.3.0

Scenario 3: Breaking Change

Before: 1.3.0

After: 2.0.0

echo "2.0.0" > VERSION
git add VERSION
git commit -m "feat!: migrate to new API v2"
git push origin sync

Result: Image tagged as ghcr.io/username/project:v2.0.0

Docker Image Tag Examples

Single Repository

VERSION file: 1.2.3

Generated tags:

ghcr.io/username/my-app:v1.2.3
ghcr.io/username/my-app:latest

Docker Compose:

services:
  app:
    image: ghcr.io/username/my-app:v1.2.3

Troubleshooting

Workflow doesn't read VERSION

Problem: Workflow fails with "VERSION file not found"

# Ensure VERSION file exists in repository root
ls -la VERSION

# If missing, create it
echo "1.0.0" > VERSION
git add VERSION
git commit -m "Add VERSION file"
git push

Image tagged incorrectly

Problem: Image shows as v.1.0.0 instead of v1.0.0

# Remove any whitespace or invalid characters
cat VERSION | tr -d '[:space:]' > VERSION.tmp && mv VERSION.tmp VERSION
git add VERSION
git commit -m "Fix VERSION file format"
git push

Using pre-release versions

Solution: Pre-release versions are supported!

# Beta release
echo "1.0.0-beta" > VERSION

# Release candidate
echo "2.0.0-rc.1" > VERSION

# Production release
echo "2.0.0" > VERSION

Results:

  • ghcr.io/username/project:v1.0.0-beta
  • ghcr.io/username/project:v2.0.0-rc.1
  • ghcr.io/username/project:v2.0.0

Best Practices

  1. Always use semantic versioning (MAJOR.MINOR.PATCH)
  2. Keep VERSION file clean (no extra whitespace)
  3. Commit version changes separately (clear commit message)
  4. Use descriptive commit messages (helps track changes)
  5. Review changes before version bump (choose correct bump type)
  6. Test before major bumps (breaking changes need verification)
  7. Document breaking changes (in commit message or CHANGELOG)

Understanding Semantic Versioning

Component Format When to Use Example
MAJOR X.0.0 Incompatible API changes, breaking changes Removing endpoints, changing response structure
MINOR 0.X.0 New features, backward compatible Adding new endpoints, new optional parameters
PATCH 0.0.X Bug fixes, backward compatible Fixing crashes, security patches

Commit Message Convention

Follow conventional commits for clarity and consistency:

# Features
git commit -m "feat: add user authentication"

# Bug fixes
git commit -m "fix: resolve memory leak in cache"

# Breaking changes
git commit -m "feat!: migrate to new database schema"

# Version bumps
git commit -m "chore: bump version to 1.2.3"

# Documentation
git commit -m "docs: update API documentation"

# Performance improvements
git commit -m "perf: optimize database queries"
